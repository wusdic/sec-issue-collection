## 1. 项目背景与目标

### 1.1 背景

市面上的安全资讯(媒体报道、厂商周报)只回答"发生了什么攻击",无法回答商业问题:

- 哪个行业、哪类单位**真金白银**在为安全事件买单?买了多少单?
- 事故之后企业**实际采购了什么**?预算由谁批?
- 监管罚款和诉讼赔偿的**真实金额分布**是什么?
- 哪些损失场景对应我们**可以销售的产品或服务**?

### 1.2 目标

| 目标 | 说明 | 衡量方式 |
|---|---|---|
| G1 建立结构化事件库 | 每日增量采集国内安全事件,按统一字段规范入库 | 日新增事件数、字段完备率 |
| G2 损失可统计 | L1–L6 六类损失分开计量,金额区分"声称/估算/已确认" | 已确认金额占比、可统计事件占比 |
| G3 支撑产品决策 | 按行业×攻击类型×损失规模输出风险画像 | 季度产品方向分析报告 |
| G4 支撑销售打单 | 事件→可售产品/服务映射,输出商机线索 | 月度线索单产出量 |
| G5 长期跟踪 | 罚款、诉讼、整改采购等滞后信息持续回填 | 回访任务完成率 |

### 1.3 非目标(明确不做)

- 不做面向公众的资讯发布平台(内部决策使用);
- 不采集、不存储、不购买任何泄露数据样本本体(只记录元数据,见第 15 节);
- 不做实时威胁情报(IOC/TTP 只作为事件属性记录,不追求分钟级时效);
- 一期不覆盖境外企业事件(境外事件仅在涉及国内主体供应链时关联记录)。

## 2. 范围定义

### 2.1 什么算"国内安全事件"

满足以下任一条件即纳入:

1. 受害主体注册地或主要经营地在中国大陆(含其境外分支被攻击但影响境内业务);
2. 受影响用户/数据主体主要为中国境内用户;
3. 中国监管机构(网信、工信、公安、金融监管、证监等)对该事件作出通报、处罚或立案;
4. 事件发生在境外但受害方为国内企业的关键供应商,且对国内产业造成可见影响(标记为"供应链关联事件")。

### 2.2 事件类型范围

勒索软件、数据泄露、网页篡改/挂马、DDoS、业务欺诈(薅羊毛/盗刷/API 滥用)、
内部人员违规(内鬼售卖数据)、供应链投毒/第三方组件事件、钓鱼与商业邮件欺诈(BEC)、
工控/物联网事件、云配置错误导致的暴露、重大漏洞被实际利用(仅"被利用"入库,纯漏洞披露不入库)、
监管处罚事件(即使未公开技术细节,罚款本身就是可决策信息)。

### 2.3 入库门槛

- 有明确受害主体(可到单位级或至少行业+地区级);
- 有至少一个可回溯的来源链接/文号/公告编号;
- 满足"最少可发布字段集"(见 4.4)。
- 单纯的漏洞预警、无受害方的攻击趋势文章、纯营销软文**不入库**。

## 3. 核心原则

> 这些原则是硬约束,采集软件与审核流程必须强制执行。

1. **决策字段优先**:不能只采集"发生过攻击",每条事件必须尽力填充商业决策字段(损失、采购、责任、罚款)。缺失的字段要标记"未披露/待回访",不允许留空白无状态。
2. **损失分开统计**:L1–L6 六类损失各自独立字段,禁止混入一个"总损失"数拍脑袋。
3. **要求 ≠ 损失**:赎金"要求 200 万美元"绝不能写成"损失 200 万美元";诉讼"索赔 5000 万"不能写成"赔偿 5000 万";媒体估算的"或致 X 亿损失"不能写成已确认损失。
4. **确认口径**:只有**企业公告、法院判决/调解书、监管处罚决定书**明确确认的金额才能进入"已确认损失";其余一律进入"声称"或"第三方估算"通道。
5. **来源可回溯**:每个关键字段(尤其金额)必须能指到具体来源条目;来源必须存档快照,防止原文删除。
6. **事件是活的**:罚款、判决、整改采购往往滞后数月甚至一年以上,事件必须有生命周期和回访机制,不是一次性录入。
7. **合规红线**:只采集公开可得信息;不接触泄露数据本体;暗网/黑产渠道只记录"被列名"这一事实与截图证据。

## 4. 事件数据模型(字段规范)

> 机器可读版本见 `schema/event.schema.json`,标准词表见 `schema/dictionaries.yaml`。
> 字段分组编号与原始需求 16 项的对照见第 17 节附录。

### 4.1 A 组 —— 事件标识与基本面

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| A1 | event_id | string | ✅ | 全局唯一,格式 `SEC-YYYYMMDD-NNNN` |
| A2 | title | string | ✅ | 一句话标题(单位+事件类型+关键后果) |
| A3 | occurred_date | date + 精度标记 | ✅ | 发生时间;精度:日/月/季/年/未知(取披露可倒推的最早时间) |
| A4 | disclosed_date | date | ✅ | 首次公开披露时间 |
| A5 | industry | 枚举(两级) | ✅ | 一级行业(金融/医疗/制造/能源/政务/教育/交通/互联网/电信/零售…)+二级细分(如 金融→城商行) |
| A6 | region | 枚举 | ✅ | 省级 + 城市(可空);全国性事件标"全国" |
| A7 | org_type | 枚举 | ✅ | 央企/国企/上市公司/民企/外企在华/政府机关/事业单位/医院/学校/其他 |
| A8 | org_name | string | ⭕ | 单位名称;未公开时填"未披露",并填 org_size |
| A9 | org_uscc | string | ⭕ | 统一社会信用代码(实体归一主键,能查到就填) |
| A10 | org_size | 枚举 | ✅ | 特大(>1万人或营收>百亿)/大/中/小微/未知 |
| A11 | related_event_ids | array | ⭕ | 同一攻击波及多家、或同一单位多次事件的关联 |
| A12 | org_tags | 多选枚举 | ⭕ | 单位补充属性:关基运营者/等保三级以上备案/央企集团成员/持牌金融机构/数据出境业务…(与 A7 组合,一个单位可多项——合规类产品的强销售信号) |
| A13 | severity | 枚举+依据 | ✅ | 事件整体严重度:特别重大/重大/较大/一般/未定级。参照《国家网络安全事件应急预案》口径按阈值自动建议(影响用户数、确认损失、停机时长、公共影响就高定级),人工可覆盖。支撑 Top 事件榜、告警与线索优先级 |

### 4.2 B 组 —— 技术面

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| B1 | affected_systems | 多选枚举+自由文本 | ✅ | 受影响业务系统:核心生产/ERP/OA/官网/APP/数据库/工控 SCADA/云资源/支付系统/CRM… |
| B2 | attack_type | 多选枚举 | ✅ | **手段**维度(勒索软件/钓鱼/供应链…),见词表 |
| B2a | consequences | 多选枚举 | ✅ | **后果**维度(数据泄露/数据被加密破坏/业务中断/资金损失…),与手段正交。一次勒索事件=手段"勒索软件"+后果"数据被加密+业务中断(+数据泄露)"。热力图与产品映射按两维交叉统计,避免手段与后果混在一个维度失真 |
| B3 | entry_vector | 多选枚举+置信度 | ✅ | 可能入口:钓鱼邮件/VPN 或边界设备漏洞/弱口令与暴力破解/供应链/内部人员/云配置错误/未修补漏洞(附 CVE)/未知 |
| B4 | vulnerability_ids | array | ⭕ | 涉及 CVE/CNVD/CNNVD 编号 |
| B5 | attacker_attribution | string+置信度 | ⭕ | 攻击组织归因(如勒索家族名);只记录来源声称,不自行归因 |
| B6 | ttp_tags | array | ⭕ | ATT&CK 技术标签(有公开分析报告时填) |
| B7 | root_cause | 枚举+文本 | ✅ | 事故原因(管理类:无备份/未打补丁/权限失控/外包失管;技术类:0day/配置错误;人为:误操作/内鬼)——允许"未披露" |
| B8 | affected_products | array | ⭕ | **涉事 IT 产品**:厂商+产品名+版本(如被利用漏洞所在的 OA/VPN/邮件系统具体产品)。销售价值极高:同产品的其他用户即为即时潜客清单("同款预警"线索);产品价值:技术栈风险画像 |
| B9 | security_controls | array | ✅ | **事发时安全控制状态**:逐项记录 {控制类别(备份/EDR/MFA/日志审计…), 状态: 缺位 / 在位但失效 / 在位被绕过 / 不明, 说明}。"无备份"和"有备份但被加密"是两个不同的产品机会——这是产品方向分析的核心输入,允许整体"不明" |

### 4.3 C 组 —— 影响与损失(商业决策核心)

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| C1 | downtime_hours | number+精度 | ✅ | 停机/停产时长(小时);未披露填 null 并标记状态 |
| C2 | loss_L1 | 金额对象 | ✅ | 直接资金/欺诈/盗窃损失 |
| C3 | loss_L2 | 金额对象 | ✅ | 停机停产、交易失败造成的收入/产量损失(产量损失可用实物量+折算金额双记录) |
| C4 | loss_L3 | 金额对象 | ✅ | 恢复、取证、法律顾问、用户通知成本 |
| C5 | loss_L4 | 金额对象 | ✅ | 罚款、赔偿、诉讼(含处罚文号、案号) |
| C6 | loss_L5 | 金额对象+文本 | ✅ | 客户流失、供应链资格取消、产品/上市延期(可无法金额化,用事实描述) |
| C7 | loss_L6 | 文本+枚举 | ✅ | 公共服务中断、物理生产破坏、人身安全影响(等级:无/轻/中/重) |
| C8 | ransom | 对象 | ✅(勒索类) | {要求金额, 币种, 是否谈判, 是否支付, 实际支付金额, 依据来源}——**要求与支付严格分列** |
| C9 | affected_users | number+精度 | ✅ | 受影响用户数(声称/估算/确认三态) |
| C10 | affected_records | number+类型 | ✅ | 泄露/受影响数据量(条数)+数据类型(身份证/手机号/病历/征信/生物特征…) |
| C11 | secondary_impact | 多选枚举 | ✅ | 是否影响:客户合同/供应链资格/牌照或资质/上市进程/信用评级/保险费率 |
| C12 | insurance | 对象 | ⭕ | 是否有网络安全保险、赔付金额(如披露) |

### 4.4 D 组 —— 响应、整改与商业情报

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| D1 | disclosure_channel | 多选枚举 | ✅ | 披露渠道:监管通报/上市公告/企业自披露/媒体曝光/攻击者泄露站/判决文书 |
| D2 | regulatory_actions | array | ⭕ | 监管动作明细:机关、依据条款原文、**规范化法规编码(legal_bases 词表:网安法/数安法/个保法/关基条例/等保规定/行业规章…)**、文号、处罚内容、日期。编码化后才能聚合统计"哪部法在驱动罚款",支撑合规产品方向分析 |
| D3 | remediation_actions | array | ✅ | 事故后**实际**整改与采购:采购品类(EDR/备份/灾备/托管服务…)、供应商(如披露)、金额(如披露)、依据来源。传闻标记为"传闻" |
| D4 | accountability | 对象 | ✅ | 谁承担责任(CIO/CISO/运维负责人/外包商;是否免职/处分/追刑)+ 谁批准整改预算(董事会/IT 部/集团)——允许"未披露" |
| D5 | sellable_mapping | 多选枚举 | ✅ | 可对应销售的我方产品/服务(由产品映射表自动初筛+人工确认,见 12.2) |
| D6 | timeline | array | ⭕ | 关键时间线:入侵→发现→披露→恢复→处罚 |
| D7 | incumbent_vendors | array | ⭕ | **事发前在用的安全产品/服务商**(仅公开来源:历史招标、厂商案例宣传),每条带证据等级。与 D3(事故后采购中标方)构成竞争分析的完整输入输出。⚠️ 仅限内部竞争分析,严禁对外用于诋毁性宣传(反不正当竞争法红线,见第 15 节) |

## 5. 损失分类与计量规则(L1–L6)

| 类别 | 定义 | 计入示例 | 不计入示例 |
|---|---|---|---|
| **L1 直接资金损失** | 被直接盗取/欺诈转出的资金 | 账户盗刷、BEC 转账被骗、加密货币被盗、薅羊毛套现 | 赎金**要求**金额;股价下跌市值蒸发 |
| **L2 收入/产量损失** | 因停机、停产、交易失败少赚的钱 | 产线停工 3 天×日产值、电商大促宕机订单损失、门诊停诊 | 灾备设备折旧;与事件无关的业绩下滑 |
| **L3 响应处置成本** | 恢复、取证、法律、通知支出 | 应急响应服务费、数据恢复、律师费、短信/信函通知用户成本、公关费用 | 事后新采购的安全产品(计入 D3 整改采购,是商机不是损失) |
| **L4 罚款赔偿诉讼** | 监管罚款、民事赔偿、和解金 | 网信办罚款、法院判赔、集体诉讼和解**已生效**金额 | 起诉**索赔**金额(未判);计提但未确认的头寸 |
| **L5 商业机会损失** | 客户流失、资格取消、延期 | 大客户解约、被剔除供应商名录、IPO 中止、产品上市推迟 | 无法归因于该事件的常规客户流失 |
| **L6 公共与物理影响** | 公共服务、物理生产、人员安全 | 城市服务中断、设备物理损坏、医疗延误伤害 | —(此类以事实与严重度等级记录为主,金额为辅) |

**计量硬规则:**

1. 每类损失是独立的"金额对象":`{声称金额, 第三方估算金额, 已确认金额, 币种, 折算人民币, 依据来源ID, 备注}`;三个通道互不覆盖。
2. **赎金单列**(C8),永不计入 L1;仅当有可信来源确认"已支付"时,支付金额计入 L1,且备注"赎金支付"。
3. 诉讼索赔记入 L4 的"声称"通道;判决/调解生效后金额才进入"已确认"通道,并记录案号。
4. 巨额结算头寸、计提准备金、市值波动一律不得写入任何"已确认损失",只能进备注。
5. 产量损失允许用实物量记录(如"停产 4.2 万吨"),折算金额需注明折算方法与单价来源。
6. 汇总统计时,默认口径只汇总"已确认"通道;"声称/估算"口径需显式选择并在报表标注。

## 6. 金额确认三态与换算规则

| 通道 | 认定标准 | 典型来源 |
|---|---|---|
| **已确认 (confirmed)** | 企业公告/年报、法院生效判决或调解书、监管处罚决定书明确载明 | 上市公司公告、裁判文书网、处罚决定书 |
| **第三方估算 (estimated)** | 研究机构、保险公司、行业协会有方法论的测算 | 厂商报告、保险精算 |
| **声称 (claimed)** | 攻击者要求、原告索赔、媒体转述、匿名爆料 | 勒索信、起诉状、新闻报道 |

换算规则:外币按**事件披露日**中国人民银行中间价折算人民币,原币种与汇率一并保存;
区间金额(如"损失 500–800 万")保留区间,统计时用区间下限做保守口径、并存中值口径。

## 7. 来源体系与可信度分级

### 7.1 来源清单(采集软件的源注册表初始配置)

**S1 官方权威(可直接支撑"已确认"字段):**

- 国家网信办及各省市网信办行政处罚公示、约谈通报
- 工信部及各省通信管理局通报(App 通报、网络安全威胁通报)
- 公安部"净网"专项通报、各地公安网安部门案件通报
- 国家网络与信息安全信息通报中心、CNCERT/CC 周报与专报
- 金融监管:人民银行、金融监管总局、证监会处罚公示
- 沪深北交易所上市公司公告(风险提示、停牌、重大事项)、港交所公告(内地业务主体)
- 裁判文书网、人民法院公告网(民事赔偿、刑事判决:非法获取计算机信息系统数据罪、侵犯公民个人信息罪等)
- 信用中国 / 各级行政处罚双公示平台
- 国务院及部委、地方政府应急通报(涉公共服务中断)

**S2 企业自披露:**

- 企业官网/公众号公告、App 站内通知、客服声明、年报 ESG 章节

**S3 行业与专业媒体、厂商报告:**

- FreeBuf、安全客、嘶吼、安全牛、E 安全等安全媒体
- 财新、界面、21 世纪经济报道等财经媒体的调查报道
- 奇安信、360、绿盟、安恒、深信服等厂商应急响应年报/专项报告
- 威胁情报社区(微步在线 X 社区等)公开分析

**S4 需佐证渠道(只作线索,不单独支撑关键字段):**

- 微博/知乎/脉脉/微信群爆料、短视频平台
- 勒索组织泄露站(leak site)对国内企业的列名监测(如 Ransomware.live 聚合)
- 泄露数据交易论坛的"售卖帖"元数据(标题、声称数据量、截图)——**只记录元数据,严禁下载样本**
- 境外媒体与研究者(DataBreaches.net 等)涉华报道

### 7.2 可信度使用规则

- 金额类字段进入"已确认"通道 → 必须挂 S1 来源(企业自披露的支出类金额可用 S2,标注);
- S4 单源事件 → 整体可信度最高只能标"单源待证",进入"待证池",不进入统计默认口径;
- 同一字段多源冲突 → 高等级来源覆盖低等级;同级冲突记录双值并标"矛盾待核";
- 每条来源必须按 7.3 规范做**完整原文存档**,原文被删除时事件仍可全文回溯。

### 7.3 原文完整存档规范(硬性要求)

> 目标:每条来源的**完整原始原文**(含正文、图片、附件)+ **对应链接地址**永久留存本地,
> 网上删帖后库内仍有与原文一致的完整副本。存档在采集时**即时**执行(延后必丢)。

**① 存档降级链**(逐级尝试,直到成功;实际达到的层级记入 archive.status):

| 层级 | 内容 | 说明 |
|---|---|---|
| L-A 完整原文 | 正文 HTML + 页面内**全部图片**、CSS、附件下载到本地,链接改写为本地相对路径,可离线原样打开 | 首选。同时保存提取后的纯文本(供检索与抽取) |
| L-B 单文件归档 | MHTML/单文件 HTML(资源内嵌)+ 整页 PDF | L-A 资源下载不全时补充 |
| L-C 快照图片 | **整页截图**;超长页面按屏高分段截成**多张图片**(编号 001、002…,相邻段重叠 10% 防跨段截断),另存一张全页拼接长图 | 反爬严重、动态渲染无法落 HTML 时的兜底,满足"不能存原文就快照存图,多页存多图" |
| L-D 失败登记 | 记录失败原因,进人工补档队列 | 不允许无声失败 |

**② 必存内容清单**:

- 原文 URL(含最终跳转后的真实地址)+ 抓取时间戳;
- 正文全部**图片**(微信公众号等平台图片有防盗链且链接会过期,必须在采集当刻带正确 Referer/UA 下载原图);
- 页面内链接的**附件**(处罚决定书 PDF、通报附件、判决文书等)一并下载;
- 页面元数据:标题、作者/发布方、发布时间、抓取时的 HTTP 状态。
- 视频只记录 URL 与封面截图,不下载(体积与版权考虑,可配置例外)。

**③ 存储结构与完整性**:

```
archive/{YYYY}/{MM}/{snapshot_id}/
  ├─ manifest.json   # URL、时间戳、层级、文件清单、每文件 SHA-256
  ├─ page.html / page.mhtml / page.pdf
  ├─ text.txt        # 提取后纯文本
  ├─ assets/         # 图片与 CSS(001.jpg …)
  ├─ attachments/    # 附件原件
  └─ screenshots/    # full.png + part_001.png, part_002.png …
```

- 每个文件计算 SHA-256 写入 manifest,存档后**只读**(WORM 策略),任何修改视为违规;
- 每月抽样 5% 校验哈希与可打开性,损坏即从冗余副本恢复(对象存储双副本);
- 事件正式库中来源的 snapshot_id ↔ 存档目录一一对应,复核台一键打开离线原文。

**④ 保留与容量**:已发布事件的来源存档**永久保留**;粗筛淘汰的候选原文保留 90 天(可配置)。
按均值 3–5 MB/篇、每日候选 50 篇估算,年增量 < 100 GB,成本可忽略,不做压缩取舍。

**⑤ 去重后薄存(节省存储)**:同一事件的转载/重复副本不重复保存完整原文——
同稿簇**首发保存完整原文**(L-A,含图片、附件),其余**转载副本只薄存正文文本+HTML**
(数 KB,而非数 MB),manifest 记录指向首发全量存档。既保留全部来源与转载计数(热度信号),
又把存储降到最低。判定用两条防线:文档层 SimHash 同稿簇 + 转载声明溯源(7.4),任一命中即薄存。

### 7.4 公众号作为一等数据源(方法论)

> 公众号本质是**"别人已做完信息筛选与事实梳理的二手加工品"**——一线安全从业者、
> 厂商、机构把散落的事件线索核实、汇总、成文,我们等于白捡这份人力。用好它的前提是
> 解决两件事:**主体定级**与**转载溯源**。配置见 `config/wechat_accounts.yaml`。

**① 每个公众号 = 一个可定级的独立数据源。** 不能把公众号笼统打成 S4:

| 主体类型 | 定级 | 说明 |
|---|---|---|
| 官方机构号(网络安全通报中心、CNVD、公安网安等) | S1 | 内容等同官方通报,可支撑已确认字段 |
| 受害企业官方号 | S2 | 自披露声明 |
| 专业媒体/厂商研究号(安全内参、FreeBuf、奇安信等) | S3 | 有编辑核实的原创报道 |
| 其它未核实号 | S4 | 需佐证,默认 |
| 营销/搬运/标题党号 | 黑名单 | 命中即丢弃,不入库、不计源发现证据 |

采集到公众号文章时,系统按发布号主体从名录**重定级**可信度(官方号自动升到 S1/S2),
而非停留渠道级 S4——这直接决定该信息能否进入"已确认"金额通道。名录持续维护:
新号先 S4,经源发现评分 + 人工确认后升级。

**② 转载必须溯源到原始出处。** 公众号互相转载极普遍,拿到的常是二手转述:

- 识别转载声明("转载自/来源:XX 公众号""原文链接:…"),提取原始账号与原文链接;
- 转载版**不作首发**(`is_primary=False`),原始出处登记为候选源(D2),推动系统直接去采原始号;
- 命中"原创声明/未经授权禁止转载"则判为首发,不误标;
- 同稿簇去重(10.2)+ 转载声明双重保障:同一事件多号转载只保留一条,并尽量追到最早原创版。

**③ 发现新号。** 文章尾部互推名单、"白名单转载"声明中的账号自动进候选源池(D3),
源池持续生长——见 9.3。这样"很多人已经帮我们做完筛选"的红利被系统化地持续吸收。

**④ 广采薄存,宁滥毋缺。** 公众号数量庞大、各有侧重覆盖面,应**尽量广泛采集**(蚂蚁搬家):
不因单篇可能重复就在前端把号丢掉——绝大多数号都能贡献一些数据,**总量上来,数据源覆盖更全**。
质量由后端保证:三层去重消化重复,转载副本按 7.3⑤ 薄存(几乎不占空间),
**去重后沉淀下来的首发原创集信息密度最高、最有价值**。所以策略是:主体一律保留并定级(不丢),
黑名单只留给确证的垃圾号;存储靠"去重后再存"而非"前端少采"来省。

## 8. 事件生命周期管理

罚款、判决、整改采购平均滞后事件曝光 3–18 个月,因此:

```
新建(draft) → 已复核(published) → 跟踪中(monitoring) → 已关闭(closed)
                                   ↑ 任何新信息可重新打开
```

- **跟踪中**:凡满足任一条件即挂回访任务——损失字段存在"声称/估算"未落地、监管已立案未处罚、诉讼进行中、整改采购未知;
- **回访节奏**:T+30 / T+90 / T+180 / T+365 天自动生成回访任务,定向检索该单位名称+处罚/判决/招标关键词;
- **回填联动**:回访重点检索**招标采购网站**(中国政府采购网、各省公共资源交易平台、军队/央企采购平台)中该单位事件后 12 个月内的安全类采购,回填 D3(事故后实际采购)——这是最直接的商机验证数据;
- **关闭条件**:所有金额通道落定或 24 个月无新信息(可配置)。

## 9. 采集与处理流程(解决方案)

### 9.1 日处理流水线

```
① 采集 → ② 粗筛 → ③ 抽取 → ④ 归一/去重 → ⑤ 人工复核 → ⑥ 入库 → ⑦ 生命周期跟踪 → ⑧ 分析输出
```

| 阶段 | 内容 | 自动化程度 |
|---|---|---|
| ① 采集 | 按源注册表定时抓取(RSS/网页爬虫/公众号采集/公告接口/泄露站监测);全文+快照落地原始库 | 全自动,每日多轮 |
| ② 粗筛 | 分类模型判定"是否国内安全事件候选",过滤漏洞预警/营销文 | 自动+置信度阈值 |
| ③ 抽取 | LLM 按字段规范做结构化抽取(输出严格对齐 schema),自动做金额三态判别(要求/索赔/判决关键词规则+模型) | 自动,产出"待复核事件" |
| ④ 归一/去重 | 实体归一(单位名→统一社会信用代码)、跨源同事件合并、与存量库比对(同单位+近时间+同类型) | 自动+疑似项转人工 |
| ⑤ 人工复核 | 审核台逐字段确认,重点核金额通道归属与来源等级;双人复核金额类字段 | 人工(工作台支撑) |
| ⑥ 入库 | 发布进正式库,写变更日志 | 自动 |
| ⑦ 跟踪 | 回访任务调度、增量信息合并 | 自动派单+人工确认 |
| ⑧ 输出 | 看板、月报、线索单、API | 自动生成+人工编审 |

### 9.2 人力配置建议(一期)

- 1 名数据运营(源维护+复核),0.5 名安全分析师(疑难事件定性、产品映射确认);
- 预期处理量:粗筛后候选 20–50 条/日,真正入库 3–10 条/日(国内公开披露密度决定)。

### 9.3 源发现与覆盖保障(查全)

> 目标:不依赖人工想到哪个网站才加哪个网站,系统**持续从网上发现新的可信源**,
> 并能度量"我们漏了多少"。

**① 两类源的不同策略:**

- **页面型源**(有固定栏目/RSS 的网站):注册后定时抓取——现 M1/M2 已覆盖;
- **搜索型源**(内容藏在平台内部,必须用查询才能触达):搜索引擎(百度/必应/360)、
  搜狗微信(公众号文章)、微博搜索、裁判文书网、信用中国、政府采购网、招投标平台。
  采集软件必须支持 **query 型适配器**:按关键词矩阵每日执行检索,而不是抓固定页面。

**② 每日定题搜索层(既找事件也找新源):**

- 关键词矩阵 = 事件词(勒索/被黑/数据泄露/网络攻击/系统瘫痪/停产/被罚/信息泄露/内鬼)
  × 行业词(银行/医院/车企/电厂…)× 后果词(罚款/判决/赔偿/停牌)× **时间限定(近 24h/近 7 天)**;
- 命中结果双重利用:正文进事件流水线;**发布站点进"候选源池"**;
- 关键词矩阵版本化管理,每月根据漏报分析调整。

**③ 引用挖掘(顺藤摸瓜):**

- 已入库文章中引用/转载的原始出处自动提取,溯源到首发站点并注册为候选源;
- 厂商年报、CNCERT 周报等汇总型文献中提到的事件,反查其原始披露渠道。

**④ 候选源生命周期(自动评估+人工把关):**

```
候选(自动发现) → 试运行(抓取 N=14 天) → 正式(入注册表) → 降级 → 退役
```

- 试运行期自动评分:原创率(非转载占比)、事件密度、与已知事实的历史准确性、更新频率、存活稳定性;
- 评分达标 → 建议转正(人工确认可信度等级 S1–S4);连续 90 天零产出或失效 → 建议退役;
- 源注册表记录每个源的"贡献度"(首发事件数/独家字段数),数据驱动源投入。

**⑤ 覆盖率对标(度量"漏了多少"):**

- 每月用**基准清单**反查:CNCERT 通报、主要厂商应急响应年报/季报、监管处罚汇总、
  法院已生效判决抽样——清单中的事件在库内的命中率即召回率估计;
- 漏报事件逐条归因(缺哪类源?关键词矩阵盲区?粗筛误杀?),形成下月源扩充与关键词调整清单;
- 漏报率是一级 KPI(见 16.1)。

**⑥ 能力边界(如实说明):** S1 官方源是有限集合,可以枚举穷尽,靠人工+引用挖掘一次性建全;
"自动发现"的主要价值在 S3/S4 长尾(自媒体、地方媒体、新爆料渠道)。
候选源转正必须人工确认可信度等级,不做全自动转正——可信度判断是本库的生命线。

### 9.4 时效保障(查新)

- **分级抓取频率**:A 级源(监管公示、高产安全媒体、搜索型源的 24h 查询)每 2–4 小时;
  B 级源每日;C 级低产源每周。级别由源贡献度自动建议、人工确认;
- **突发触发**:微博热搜、百度热榜按关键词监测,命中安全事件词即触发全源即时抓取一轮;
- **增量抓取**:RSS/sitemap 自动发现;页面内容哈希比对,未变更不重复下载;
- **时效 KPI**:公开披露 → 入库(待复核)中位时延 ≤ 24 小时,S1 监管公示 ≤ 12 小时;
  每条事件记录 `disclosed_date → 入库时间` 差值用于考核。

## 10. 去重与实体归一(查重:三层去重)

> 国内媒体转载率极高,同一篇稿件常见数十个站点转发;同一事件又会在数月后因
> 处罚/判决被"二次报道"。去重必须在三个层面分别做,任何一层缺失都会造成重复。

### 10.1 第一层:URL 去重(抓取前)

- URL 归一化(去除跟踪参数、统一协议与大小写)后查已抓集合(布隆过滤器+精确表);
- 命中即跳过下载,只累计"再次出现"计数(热度信号)。

### 10.2 第二层:文档去重(抓取后、抽取前)

- 正文提取后计算 SimHash/MinHash,近重复(转载、洗稿微调)聚合为**同稿簇**;
- 同稿簇只走一次抽取流水线,保留**首发**版本为主文档,其余仅记来源计数与最早发布时间;
- 首发判定:发布时间最早 + 站点原创标记;转载量作为事件热度指标存档。

### 10.3 第三层:事件去重(入库时与入库后)

- **单位主键**:统一社会信用代码;无法获取时用"标准化名称+地区"临时键,后续合并;
- **事件指纹**:单位键 + 事件类型 + 发生时间窗(±14 天)→ 命中即提示合并;
- **语义相似召回**:指纹未命中时,用事件摘要 embedding 与存量库近邻检索兜底
  (覆盖"单位名未披露但描述指向同一事件"、"表述差异大"的情况),疑似项转人工;
- **跨时间合并**:数月后出现的处罚决定、判决书、整改招标报道,必须并回原事件
  (更新 C5/D2/D3 字段并推进生命周期),**不得新建事件**——回访任务与语义召回共同保障;
- **一波多家**:同一漏洞/同一团伙波及多家单位时,每家一条事件,用 `related_event_ids` 互链,并建"事件簇"视图;
- **合并规则**:合并保留全部来源,字段取可信度更高的值,全程写变更日志;
- **查重 KPI**:发布库重复事件率 ≤ 1%(月度抽检),同稿簇聚合率纳入流水线监控。

## 11. 质量控制体系

1. **双人复核**:金额字段、监管处罚字段必须录入人+复核人双签;
2. **完备度评分**:系统按字段权重自动打分(决策字段权重高),低于阈值的事件进"补采队列";
3. **抽样审计**:每月抽 10% 已发布事件溯源核对,错误率>2% 触发流程整改;
4. **口径回归测试**:统计报表发布前自动校验——任何"已确认"金额必须能追到 S1/S2 来源 ID,否则报表拒绝生成;
5. **词表治理**:行业、攻击类型、产品映射词表版本化管理,变更需评审(否则历史统计口径漂移)。

## 12. 商业应用层

### 12.1 分析视图(一期看板)

- 行业×攻击类型热力图(近 90 天/近 12 个月);
- L1–L6 损失分布(按行业/地区/单位类型,已确认口径与估算口径双开关);
- 监管处罚榜:处罚机关、依据条款、金额分布 → 合规类产品的需求信号;
- 停机时长分布 → 灾备/备份/应急响应服务的需求信号;
- 事故后采购追踪:事件→招标记录命中率、采购品类分布 → 最直接的市场验证;
- **控制缺失统计**(B9):各行业"缺位 / 在位但失效"的控制分布 → 建设类产品 vs 效果类产品的方向判断;
- **涉事产品榜**(B8):被利用最多的 IT 产品/厂商排名 → 同款预警线索池与技术栈风险画像;
- **竞争看板**(D7+D3):事发前在位安全厂商 ↔ 事故后中标厂商的流转矩阵 → 替换机会与竞品得失;
- 待证池看板:S4 线索转化率。

### 12.2 产品/服务映射表(初版,需按公司产品线校准)

> 映射规则引擎(M8)按 **手段(B2)× 入口(B3)× 根因(B7)× 后果(B2a)** 四维组合触发,
> 不是只看攻击类型单维;下表为按主信号简化的初版。

| 事件信号 | 可售产品/服务 |
|---|---|
| 勒索软件、备份缺失 | 备份与恢复、灾备、EDR、勒索防护评估、应急响应服务 |
| 数据泄露、内鬼售卖 | 数据库审计、DLP、零信任/权限治理、数据分类分级服务 |
| 钓鱼/BEC | 邮件安全网关、安全意识培训、MFA |
| 边界设备/VPN 漏洞入侵 | 攻击面管理、漏洞管理、渗透测试、SASE |
| 供应链事件 | 软件供应链安全(SBOM)、供应商安全评估 |
| 云配置错误 | CSPM、云安全托管 |
| 工控事件 | 工控安全审计、网络隔离、OT 监测 |
| 监管处罚(等保/个保) | 等保测评辅导、个保合规咨询、密评 |
| 任何重大事件 | MSS 托管安全服务、网络安全保险经纪合作 |

### 12.3 线索评分与采购窗口模型(销售侧核心)

**线索优先级评分**(M8 规则引擎自动计算,0–100):

```
线索分 = 严重度(A13) × 采购窗口阶段权重 × 产品匹配度(D5 命中数) × 单位规模(A10) × 可触达性
```

- **可触达性**加分项:同省有销售团队、org_tags 命中我方优势客群(如持牌金融机构)、
  同集团已有合作、affected_products 命中我方可替代产品;
- 评分随时间衰减,窗口期外自动降级归档。

**采购窗口三阶段**(事件披露日起算,线索单按阶段给出不同打法建议):

| 阶段 | 时间 | 客户状态 | 切入产品/话术 |
|---|---|---|---|
| 应急期 | 0–1 月 | 救火,预算特批 | 应急响应、取证、临时管控;禁忌:趁火打劫式推销 |
| 整改期 | 1–6 月 | 复盘定方案,监管要求整改 | 对应 security_controls 缺口的建设类产品;整改报告要求是最强抓手 |
| 预算期 | 6–18 月 | 纳入下年度预算,走招标 | 大额平台类项目;紧盯政府采购意向公开与招标预告(M6 回访联动) |

**同款预警线索**(利用 B8):某产品漏洞被利用的事件入库后,自动生成"该产品其他已知用户"
潜客清单(用户名单来自公开招标历史与厂商案例,P3 企业库集成后自动化),
话术依据是已发生的同款事件——这是转化率最高的线索类型。

**无映射事件 = 产品空白清单**(产品侧):D5 为空或映射牵强的事件自动进"白区清单",
季度产品方向报告必须逐条回答:是市场太小不做,还是产品缺口该补——让"我们没产品接"的机会显性化。

### 12.4 配套数据集(事件库之外必须补齐的三块)

> 如实说明:**只靠事件库撑不起完整的产品方向决策**。国内安全市场需求相当程度由
> 政策驱动(等保、密评、数据出境评估、专项行动),且市场规模要看采购全量而非事件样本。
> 事件库是"痛点信号",还需三个轻量配套数据集:

| 数据集 | 内容 | 用途 | 建设方式 |
|---|---|---|---|
| ① 政策监管动态库 | 新法规规章、征求意见稿、专项行动(如打击侵犯公民个人信息专项)、标准发布(等保/密评/行标)、监管约谈风向 | 需求预判:政策落地前 6–12 个月布局对应产品;线索话术的合规依据 | 复用 M2 采集引擎,源为部委官网+行业协会,量小,结构简单(政策条目+生效时间+影响行业+对应产品) |
| ② 招标采购全量集 | 安全类招标/中标全量(不只事件后单位):品类、金额、采购方、中标方 | 市场规模(TAM)与增速测算、竞品份额与价格带、区域热度 | 复用 M2 搜索型适配器(政府采购网、公共资源交易平台),按 procurement_categories 打标 |
| ③ 企业名录库 | 行业×规模×地区×org_tags 的单位名录 | 同行业同规模潜客扩展、同款预警名单 | P3 集成外部工商数据(企查查类),不自建 |

三库与事件库通过单位主键(统一社会信用代码)和产品词表打通,
产品方向季报的标准结构 = 事件痛点信号(事件库)× 政策驱动(①)× 市场资金流向(②)。

### 12.5 输出物

- **月度行业风险简报**(内部):Top 事件、损失统计、监管动向、产品需求信号;
- **销售线索单**(周度):按 12.3 评分排序,含采购窗口阶段、建议切入产品、事件话术依据、同款预警名单;
- **行业案例话术包**(季度更新):按行业整理的脱敏事件故事集(同行出了什么事、损失多少、被罚多少、后来买了什么)——一线销售直接可用的"恐惧+合规+同行"三类论据,全部标注公开来源;
- **季度产品方向报告**:事件痛点(含控制缺失统计与白区清单)× 政策驱动 × 招标资金流向 三源交叉的投入建议;
- **API/导出**:供 CRM 打通,线索自动挂到销售管道,并回流跟进结果(见 16.1 转化率 KPI)。

- **部署**:一期单机 Docker Compose 即可,日处理量很小;重点投入在词表、schema 与复核流程而非算力;
- **扩展预留**:事件簇图谱(单位—事件—攻击组织—供应商关系)、与 CRM/工单系统对接。

## 15. 合规与法律边界

1. 仅采集**公开渠道**信息;不绕过登录、付费墙及访问控制;遵守目标站 robots 与频率礼仪;
2. **严禁**下载、存储、购买、传播任何泄露数据样本;泄露论坛/暗网只记录帖子的元数据与截图证据;
3. 事件库含企业负面信息,**仅限内部使用**;对外引用需经法务审核,避免不正当竞争与名誉权风险;销售话术引用事件时只引用公开来源事实,不做未证实指控;
3a. **竞争情报红线**:D7(在位厂商)与 D3(中标厂商)数据仅限内部竞争分析与产品复盘;
    严禁在销售材料、公开场合以"某厂商产品导致出事"等方式贬损竞品(《反不正当竞争法》商业诋毁条款);
    "同款预警"线索话术只引用产品漏洞公告与公开事件事实,不评价厂商;
4. 涉及个人信息的字段(如责任人姓名)只记录公开来源已披露的职务信息,遵循最小必要;
5. 数据库自身按《数据安全法》《个人信息保护法》做好访问控制、审计与分级;它本身也是敏感资产;
6. 使用境外来源(泄露站聚合等)时注意跨境访问合规,由公司法务出具采集渠道白名单。
