# 国内行业安全事件库 —— 需求与解决方案(v1.0)

## 1. 项目背景与目标

### 1.1 背景

市面上的安全资讯(媒体报道、厂商周报)只回答"发生了什么攻击",无法回答商业问题:

- 哪个行业、哪类单位**真金白银**在为安全事件买单?买了多少单?
- 事故之后企业**实际采购了什么**?预算由谁批?
- 监管罚款和诉讼赔偿的**真实金额分布**是什么?
- 哪些损失场景对应我们**可以销售的产品或服务**?

### 1.2 目标

| 目标 | 说明 | 衡量方式 |
|---|---|---|
| G1 建立结构化事件库 | 每日增量采集国内安全事件,按统一字段规范入库 | 日新增事件数、字段完备率 |
| G2 损失可统计 | L1–L6 六类损失分开计量,金额区分"声称/估算/已确认" | 已确认金额占比、可统计事件占比 |
| G3 支撑产品决策 | 按行业×攻击类型×损失规模输出风险画像 | 季度产品方向分析报告 |
| G4 支撑销售打单 | 事件→可售产品/服务映射,输出商机线索 | 月度线索单产出量 |
| G5 长期跟踪 | 罚款、诉讼、整改采购等滞后信息持续回填 | 回访任务完成率 |

### 1.3 非目标(明确不做)

- 不做面向公众的资讯发布平台(内部决策使用);
- 不采集、不存储、不购买任何泄露数据样本本体(只记录元数据,见第 15 节);
- 不做实时威胁情报(IOC/TTP 只作为事件属性记录,不追求分钟级时效);
- 一期不覆盖境外企业事件(境外事件仅在涉及国内主体供应链时关联记录)。

## 2. 范围定义

### 2.1 什么算"国内安全事件"

满足以下任一条件即纳入:

1. 受害主体注册地或主要经营地在中国大陆(含其境外分支被攻击但影响境内业务);
2. 受影响用户/数据主体主要为中国境内用户;
3. 中国监管机构(网信、工信、公安、金融监管、证监等)对该事件作出通报、处罚或立案;
4. 事件发生在境外但受害方为国内企业的关键供应商,且对国内产业造成可见影响(标记为"供应链关联事件")。

### 2.2 事件类型范围

勒索软件、数据泄露、网页篡改/挂马、DDoS、业务欺诈(薅羊毛/盗刷/API 滥用)、
内部人员违规(内鬼售卖数据)、供应链投毒/第三方组件事件、钓鱼与商业邮件欺诈(BEC)、
工控/物联网事件、云配置错误导致的暴露、重大漏洞被实际利用(仅"被利用"入库,纯漏洞披露不入库)、
监管处罚事件(即使未公开技术细节,罚款本身就是可决策信息)。

### 2.3 入库门槛

- 有明确受害主体(可到单位级或至少行业+地区级);
- 有至少一个可回溯的来源链接/文号/公告编号;
- 满足"最少可发布字段集"(见 4.4)。
- 单纯的漏洞预警、无受害方的攻击趋势文章、纯营销软文**不入库**。

## 3. 核心原则

> 这些原则是硬约束,采集软件与审核流程必须强制执行。

1. **决策字段优先**:不能只采集"发生过攻击",每条事件必须尽力填充商业决策字段(损失、采购、责任、罚款)。缺失的字段要标记"未披露/待回访",不允许留空白无状态。
2. **损失分开统计**:L1–L6 六类损失各自独立字段,禁止混入一个"总损失"数拍脑袋。
3. **要求 ≠ 损失**:赎金"要求 200 万美元"绝不能写成"损失 200 万美元";诉讼"索赔 5000 万"不能写成"赔偿 5000 万";媒体估算的"或致 X 亿损失"不能写成已确认损失。
4. **确认口径**:只有**企业公告、法院判决/调解书、监管处罚决定书**明确确认的金额才能进入"已确认损失";其余一律进入"声称"或"第三方估算"通道。
5. **来源可回溯**:每个关键字段(尤其金额)必须能指到具体来源条目;来源必须存档快照,防止原文删除。
6. **事件是活的**:罚款、判决、整改采购往往滞后数月甚至一年以上,事件必须有生命周期和回访机制,不是一次性录入。
7. **合规红线**:只采集公开可得信息;不接触泄露数据本体;暗网/黑产渠道只记录"被列名"这一事实与截图证据。

## 4. 事件数据模型(字段规范)

> 机器可读版本见 `schema/event.schema.json`,标准词表见 `schema/dictionaries.yaml`。
> 字段分组编号与原始需求 16 项的对照见第 17 节附录。

### 4.1 A 组 —— 事件标识与基本面

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| A1 | event_id | string | ✅ | 全局唯一,格式 `SEC-YYYYMMDD-NNNN` |
| A2 | title | string | ✅ | 一句话标题(单位+事件类型+关键后果) |
| A3 | occurred_date | date + 精度标记 | ✅ | 发生时间;精度:日/月/季/年/未知(取披露可倒推的最早时间) |
| A4 | disclosed_date | date | ✅ | 首次公开披露时间 |
| A5 | industry | 枚举(两级) | ✅ | 一级行业(金融/医疗/制造/能源/政务/教育/交通/互联网/电信/零售…)+二级细分(如 金融→城商行) |
| A6 | region | 枚举 | ✅ | 省级 + 城市(可空);全国性事件标"全国" |
| A7 | org_type | 枚举 | ✅ | 央企/国企/上市公司/民企/外企在华/政府机关/事业单位/医院/学校/其他 |
| A8 | org_name | string | ⭕ | 单位名称;未公开时填"未披露",并填 org_size |
| A9 | org_uscc | string | ⭕ | 统一社会信用代码(实体归一主键,能查到就填) |
| A10 | org_size | 枚举 | ✅ | 特大(>1万人或营收>百亿)/大/中/小微/未知 |
| A11 | related_event_ids | array | ⭕ | 同一攻击波及多家、或同一单位多次事件的关联 |

### 4.2 B 组 —— 技术面

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| B1 | affected_systems | 多选枚举+自由文本 | ✅ | 受影响业务系统:核心生产/ERP/OA/官网/APP/数据库/工控 SCADA/云资源/支付系统/CRM… |
| B2 | attack_type | 多选枚举 | ✅ | 见 2.2 事件类型词表 |
| B3 | entry_vector | 多选枚举+置信度 | ✅ | 可能入口:钓鱼邮件/VPN 或边界设备漏洞/弱口令与暴力破解/供应链/内部人员/云配置错误/未修补漏洞(附 CVE)/未知 |
| B4 | vulnerability_ids | array | ⭕ | 涉及 CVE/CNVD/CNNVD 编号 |
| B5 | attacker_attribution | string+置信度 | ⭕ | 攻击组织归因(如勒索家族名);只记录来源声称,不自行归因 |
| B6 | ttp_tags | array | ⭕ | ATT&CK 技术标签(有公开分析报告时填) |
| B7 | root_cause | 枚举+文本 | ✅ | 事故原因(管理类:无备份/未打补丁/权限失控/外包失管;技术类:0day/配置错误;人为:误操作/内鬼)——允许"未披露" |

### 4.3 C 组 —— 影响与损失(商业决策核心)

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| C1 | downtime_hours | number+精度 | ✅ | 停机/停产时长(小时);未披露填 null 并标记状态 |
| C2 | loss_L1 | 金额对象 | ✅ | 直接资金/欺诈/盗窃损失 |
| C3 | loss_L2 | 金额对象 | ✅ | 停机停产、交易失败造成的收入/产量损失(产量损失可用实物量+折算金额双记录) |
| C4 | loss_L3 | 金额对象 | ✅ | 恢复、取证、法律顾问、用户通知成本 |
| C5 | loss_L4 | 金额对象 | ✅ | 罚款、赔偿、诉讼(含处罚文号、案号) |
| C6 | loss_L5 | 金额对象+文本 | ✅ | 客户流失、供应链资格取消、产品/上市延期(可无法金额化,用事实描述) |
| C7 | loss_L6 | 文本+枚举 | ✅ | 公共服务中断、物理生产破坏、人身安全影响(等级:无/轻/中/重) |
| C8 | ransom | 对象 | ✅(勒索类) | {要求金额, 币种, 是否谈判, 是否支付, 实际支付金额, 依据来源}——**要求与支付严格分列** |
| C9 | affected_users | number+精度 | ✅ | 受影响用户数(声称/估算/确认三态) |
| C10 | affected_records | number+类型 | ✅ | 泄露/受影响数据量(条数)+数据类型(身份证/手机号/病历/征信/生物特征…) |
| C11 | secondary_impact | 多选枚举 | ✅ | 是否影响:客户合同/供应链资格/牌照或资质/上市进程/信用评级/保险费率 |
| C12 | insurance | 对象 | ⭕ | 是否有网络安全保险、赔付金额(如披露) |

### 4.4 D 组 —— 响应、整改与商业情报

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| D1 | disclosure_channel | 多选枚举 | ✅ | 披露渠道:监管通报/上市公告/企业自披露/媒体曝光/攻击者泄露站/判决文书 |
| D2 | regulatory_actions | array | ⭕ | 监管动作明细:机关、依据条款(如网络安全法第 21 条)、文号、处罚内容、日期 |
| D3 | remediation_actions | array | ✅ | 事故后**实际**整改与采购:采购品类(EDR/备份/灾备/托管服务…)、供应商(如披露)、金额(如披露)、依据来源。传闻标记为"传闻" |
| D4 | accountability | 对象 | ✅ | 谁承担责任(CIO/CISO/运维负责人/外包商;是否免职/处分/追刑)+ 谁批准整改预算(董事会/IT 部/集团)——允许"未披露" |
| D5 | sellable_mapping | 多选枚举 | ✅ | 可对应销售的我方产品/服务(由产品映射表自动初筛+人工确认,见 12.2) |
| D6 | timeline | array | ⭕ | 关键时间线:入侵→发现→披露→恢复→处罚 |

### 4.5 E 组 —— 来源与元数据

| # | 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|---|
| E1 | sources | array | ✅ | 每条:URL/文号、标题、发布方、发布时间、可信度等级(S1–S4)、快照存档 ID、支撑了哪些字段 |
| E2 | confidence_overall | 枚举 | ✅ | 事件整体可信度:已证实/多源印证/单源待证/存疑 |
| E3 | completeness_score | number | 自动 | 字段完备度得分(0–100,系统计算) |
| E4 | status | 枚举 | ✅ | 生命周期状态(见第 8 节) |
| E5 | review | 对象 | ✅ | 录入人、复核人、录入/复核时间 |
| E6 | change_log | array | 自动 | 每次修改的字段、旧值、新值、依据来源 |

**4.6 最少可发布字段集(入库门槛)**:A1–A7、A10、B1、B2、B7(可为"未披露")、C 组全部字段(值可为 null 但状态必填)、D1、E1、E2、E4。

## 5. 损失分类与计量规则(L1–L6)

| 类别 | 定义 | 计入示例 | 不计入示例 |
|---|---|---|---|
| **L1 直接资金损失** | 被直接盗取/欺诈转出的资金 | 账户盗刷、BEC 转账被骗、加密货币被盗、薅羊毛套现 | 赎金**要求**金额;股价下跌市值蒸发 |
| **L2 收入/产量损失** | 因停机、停产、交易失败少赚的钱 | 产线停工 3 天×日产值、电商大促宕机订单损失、门诊停诊 | 灾备设备折旧;与事件无关的业绩下滑 |
| **L3 响应处置成本** | 恢复、取证、法律、通知支出 | 应急响应服务费、数据恢复、律师费、短信/信函通知用户成本、公关费用 | 事后新采购的安全产品(计入 D3 整改采购,是商机不是损失) |
| **L4 罚款赔偿诉讼** | 监管罚款、民事赔偿、和解金 | 网信办罚款、法院判赔、集体诉讼和解**已生效**金额 | 起诉**索赔**金额(未判);计提但未确认的头寸 |
| **L5 商业机会损失** | 客户流失、资格取消、延期 | 大客户解约、被剔除供应商名录、IPO 中止、产品上市推迟 | 无法归因于该事件的常规客户流失 |
| **L6 公共与物理影响** | 公共服务、物理生产、人员安全 | 城市服务中断、设备物理损坏、医疗延误伤害 | —(此类以事实与严重度等级记录为主,金额为辅) |

**计量硬规则:**

1. 每类损失是独立的"金额对象":`{声称金额, 第三方估算金额, 已确认金额, 币种, 折算人民币, 依据来源ID, 备注}`;三个通道互不覆盖。
2. **赎金单列**(C8),永不计入 L1;仅当有可信来源确认"已支付"时,支付金额计入 L1,且备注"赎金支付"。
3. 诉讼索赔记入 L4 的"声称"通道;判决/调解生效后金额才进入"已确认"通道,并记录案号。
4. 巨额结算头寸、计提准备金、市值波动一律不得写入任何"已确认损失",只能进备注。
5. 产量损失允许用实物量记录(如"停产 4.2 万吨"),折算金额需注明折算方法与单价来源。
6. 汇总统计时,默认口径只汇总"已确认"通道;"声称/估算"口径需显式选择并在报表标注。

## 6. 金额确认三态与换算规则

| 通道 | 认定标准 | 典型来源 |
|---|---|---|
| **已确认 (confirmed)** | 企业公告/年报、法院生效判决或调解书、监管处罚决定书明确载明 | 上市公司公告、裁判文书网、处罚决定书 |
| **第三方估算 (estimated)** | 研究机构、保险公司、行业协会有方法论的测算 | 厂商报告、保险精算 |
| **声称 (claimed)** | 攻击者要求、原告索赔、媒体转述、匿名爆料 | 勒索信、起诉状、新闻报道 |

换算规则:外币按**事件披露日**中国人民银行中间价折算人民币,原币种与汇率一并保存;
区间金额(如"损失 500–800 万")保留区间,统计时用区间下限做保守口径、并存中值口径。

## 7. 来源体系与可信度分级

### 7.1 来源清单(采集软件的源注册表初始配置)

**S1 官方权威(可直接支撑"已确认"字段):**

- 国家网信办及各省市网信办行政处罚公示、约谈通报
- 工信部及各省通信管理局通报(App 通报、网络安全威胁通报)
- 公安部"净网"专项通报、各地公安网安部门案件通报
- 国家网络与信息安全信息通报中心、CNCERT/CC 周报与专报
- 金融监管:人民银行、金融监管总局、证监会处罚公示
- 沪深北交易所上市公司公告(风险提示、停牌、重大事项)、港交所公告(内地业务主体)
- 裁判文书网、人民法院公告网(民事赔偿、刑事判决:非法获取计算机信息系统数据罪、侵犯公民个人信息罪等)
- 信用中国 / 各级行政处罚双公示平台
- 国务院及部委、地方政府应急通报(涉公共服务中断)

**S2 企业自披露:**

- 企业官网/公众号公告、App 站内通知、客服声明、年报 ESG 章节

**S3 行业与专业媒体、厂商报告:**

- FreeBuf、安全客、嘶吼、安全牛、E 安全等安全媒体
- 财新、界面、21 世纪经济报道等财经媒体的调查报道
- 奇安信、360、绿盟、安恒、深信服等厂商应急响应年报/专项报告
- 威胁情报社区(微步在线 X 社区等)公开分析

**S4 需佐证渠道(只作线索,不单独支撑关键字段):**

- 微博/知乎/脉脉/微信群爆料、短视频平台
- 勒索组织泄露站(leak site)对国内企业的列名监测(如 Ransomware.live 聚合)
- 泄露数据交易论坛的"售卖帖"元数据(标题、声称数据量、截图)——**只记录元数据,严禁下载样本**
- 境外媒体与研究者(DataBreaches.net 等)涉华报道

### 7.2 可信度使用规则

- 金额类字段进入"已确认"通道 → 必须挂 S1 来源(企业自披露的支出类金额可用 S2,标注);
- S4 单源事件 → 整体可信度最高只能标"单源待证",进入"待证池",不进入统计默认口径;
- 同一字段多源冲突 → 高等级来源覆盖低等级;同级冲突记录双值并标"矛盾待核";
- 每条来源必须存**网页快照**(HTML+截图),原文被删除时事件仍可回溯。

## 8. 事件生命周期管理

罚款、判决、整改采购平均滞后事件曝光 3–18 个月,因此:

```
新建(draft) → 已复核(published) → 跟踪中(monitoring) → 已关闭(closed)
                                   ↑ 任何新信息可重新打开
```

- **跟踪中**:凡满足任一条件即挂回访任务——损失字段存在"声称/估算"未落地、监管已立案未处罚、诉讼进行中、整改采购未知;
- **回访节奏**:T+30 / T+90 / T+180 / T+365 天自动生成回访任务,定向检索该单位名称+处罚/判决/招标关键词;
- **回填联动**:回访重点检索**招标采购网站**(中国政府采购网、各省公共资源交易平台、军队/央企采购平台)中该单位事件后 12 个月内的安全类采购,回填 D3(事故后实际采购)——这是最直接的商机验证数据;
- **关闭条件**:所有金额通道落定或 24 个月无新信息(可配置)。

## 9. 采集与处理流程(解决方案)

### 9.1 日处理流水线

```
① 采集 → ② 粗筛 → ③ 抽取 → ④ 归一/去重 → ⑤ 人工复核 → ⑥ 入库 → ⑦ 生命周期跟踪 → ⑧ 分析输出
```

| 阶段 | 内容 | 自动化程度 |
|---|---|---|
| ① 采集 | 按源注册表定时抓取(RSS/网页爬虫/公众号采集/公告接口/泄露站监测);全文+快照落地原始库 | 全自动,每日多轮 |
| ② 粗筛 | 分类模型判定"是否国内安全事件候选",过滤漏洞预警/营销文 | 自动+置信度阈值 |
| ③ 抽取 | LLM 按字段规范做结构化抽取(输出严格对齐 schema),自动做金额三态判别(要求/索赔/判决关键词规则+模型) | 自动,产出"待复核事件" |
| ④ 归一/去重 | 实体归一(单位名→统一社会信用代码)、跨源同事件合并、与存量库比对(同单位+近时间+同类型) | 自动+疑似项转人工 |
| ⑤ 人工复核 | 审核台逐字段确认,重点核金额通道归属与来源等级;双人复核金额类字段 | 人工(工作台支撑) |
| ⑥ 入库 | 发布进正式库,写变更日志 | 自动 |
| ⑦ 跟踪 | 回访任务调度、增量信息合并 | 自动派单+人工确认 |
| ⑧ 输出 | 看板、月报、线索单、API | 自动生成+人工编审 |

### 9.2 人力配置建议(一期)

- 1 名数据运营(源维护+复核),0.5 名安全分析师(疑难事件定性、产品映射确认);
- 预期处理量:粗筛后候选 20–50 条/日,真正入库 3–10 条/日(国内公开披露密度决定)。

## 10. 去重与实体归一

- **单位主键**:统一社会信用代码;无法获取时用"标准化名称+地区"临时键,后续合并;
- **事件指纹**:单位键 + 事件类型 + 发生时间窗(±14 天)→ 命中即提示合并;攻击者泄露站列名与后续媒体报道要合并为同一事件;
- **一波多家**:同一漏洞/同一团伙波及多家单位时,每家一条事件,用 `related_event_ids` 互链,并建"事件簇"视图;
- **合并规则**:合并保留全部来源,字段取可信度更高的值,全程写变更日志。

---

## 11. 质量控制体系

1. **双人复核**:金额字段、监管处罚字段必须录入人+复核人双签;
2. **完备度评分**:系统按字段权重自动打分(决策字段权重高),低于阈值的事件进"补采队列";
3. **抽样审计**:每月抽 10% 已发布事件溯源核对,错误率>2% 触发流程整改;
4. **口径回归测试**:统计报表发布前自动校验——任何"已确认"金额必须能追到 S1/S2 来源 ID,否则报表拒绝生成;
5. **词表治理**:行业、攻击类型、产品映射词表版本化管理,变更需评审(否则历史统计口径漂移)。

## 12. 商业应用层

### 12.1 分析视图(一期看板)

- 行业×攻击类型热力图(近 90 天/近 12 个月);
- L1–L6 损失分布(按行业/地区/单位类型,已确认口径与估算口径双开关);
- 监管处罚榜:处罚机关、依据条款、金额分布 → 合规类产品的需求信号;
- 停机时长分布 → 灾备/备份/应急响应服务的需求信号;
- 事故后采购追踪:事件→招标记录命中率、采购品类分布 → 最直接的市场验证;
- 待证池看板:S4 线索转化率。

### 12.2 产品/服务映射表(初版,需按公司产品线校准)

| 事件信号 | 可售产品/服务 |
|---|---|

### 12.3 输出物

## 13. 采集软件功能需求清单

> 供下一阶段软件设计直接引用,按模块编号。

**M1 源管理**:源注册表 CRUD(URL/类型/抓取频率/解析模板/可信度等级);源健康监控(失效告警);新源试运行沙箱。

**M2 采集引擎**:定时调度;网页/RSS/公众号/公告页多适配器;全文抓取+HTML 快照+整页截图存档;代理与限速;失败重试。

**M3 智能抽取**:候选分类(是否事件);LLM 结构化抽取(严格对齐 `event.schema.json`);金额三态判别(关键词规则:"要求/索赔/拟处罚"→声称,"判决/决定书/公告确认"→需人工确认进已确认);抽取置信度输出。

**M4 归一去重**:单位名标准化+工商信息联查(统一社会信用代码);事件指纹碰撞检测;合并工作流。

**M5 复核工作台**:左原文右表单的逐字段审核界面;字段级来源绑定;双人复核流转;快捷词表选择;驳回/补采/发布操作;金额通道错误的强提示(如检测到"要求"字样却填入已确认通道时阻断)。

**M6 生命周期与回访**:状态机;T+30/90/180/365 回访任务自动生成;定向检索助手(单位名+处罚/判决/招标关键词一键查询裁判文书、信用中国、政府采购网);回填合并。

**M7 检索与看板**:多维筛选(行业/地区/类型/损失区间/时间);12.1 全部视图;待证池;导出 CSV/Excel。

**M8 线索与报告**:产品映射规则引擎(可配置);线索单生成;月报模板半自动生成;API(REST,token 鉴权)。

**M9 系统管理**:用户与角色(录入员/复核员/分析师/只读);词表版本管理;变更日志与审计;数据备份。

## 14. 系统架构建议

```
[源注册表] → [调度器] → [采集适配器集群] → [原始库: 原文+快照(对象存储)]
                                          ↓
                              [粗筛/抽取服务(LLM)] → [待复核队列]
                                          ↓
                              [复核工作台(Web)] ⇄ [事件正式库(PostgreSQL)]
                                          ↓
                  [回访调度] [统计/口径校验] [看板 BI] [线索/API 输出]
```

- **存储**:PostgreSQL(事件结构化数据,JSONB 存金额对象与来源数组);对象存储放快照;可选 Elasticsearch 做全文检索;
- **抽取**:LLM API + 规则引擎混合;抽取 prompt 与 schema 版本绑定;
- **部署**:一期单机 Docker Compose 即可,日处理量很小;重点投入在词表、schema 与复核流程而非算力;
- **扩展预留**:事件簇图谱(单位—事件—攻击组织—供应商关系)、与 CRM/工单系统对接。

## 15. 合规与法律边界

1. 仅采集**公开渠道**信息;不绕过登录、付费墙及访问控制;遵守目标站 robots 与频率礼仪;
2. **严禁**下载、存储、购买、传播任何泄露数据样本;泄露论坛/暗网只记录帖子的元数据与截图证据;
3. 事件库含企业负面信息,**仅限内部使用**;对外引用需经法务审核,避免不正当竞争与名誉权风险;销售话术引用事件时只引用公开来源事实,不做未证实指控;
4. 涉及个人信息的字段(如责任人姓名)只记录公开来源已披露的职务信息,遵循最小必要;
5. 数据库自身按《数据安全法》《个人信息保护法》做好访问控制、审计与分级;它本身也是敏感资产;
6. 使用境外来源(泄露站聚合等)时注意跨境访问合规,由公司法务出具采集渠道白名单。
