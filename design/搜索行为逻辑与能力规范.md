# 搜索行为逻辑与能力规范(v0.1)

> 本文把方案与详细设计中**所有涉及"搜索"的行为**收敛为一份统一规范:八类搜索行为(B1–B8)
> × 十项通用能力(C1–C10)。查询型适配器与调度器按此实现,任何新增搜索需求先归入 B 类,
> 复用 C 能力,不另起炉灶。
> 本次查验补齐的逻辑缺口见第 3 节(监控名单、跳转还原、水位线等 6 项)。

---

## 1. 八类搜索行为(B1–B8)

> 每类行为定义:触发时机 → 输入 → 查询构造 → 结果处置 → 产出去向。
> 所有行为的结果统一走:URL 归一化去重(10.1)→ 新 URL 进抓取存档流水线;
> 发布方提取 → 源发现证据(D1);查询执行记录入 `keyword_run`。

| # | 行为 | 触发 | 输入 | 查询构造 | 产出去向 |
|---|---|---|---|---|---|
| **B1 事件发现搜索** | 每日定时(A 级源每 2–4h) | 关键词矩阵(事件×行业×后果×单位词) | 词组 + 时间限定(近24h/7天)+ 排除词 | 候选文档 → 粗筛;发布方 → 源候选 |
| **B2 找源搜索** | 每周 | discovery.yaml 找源检索词 | "渠道类"查询词,不带时间限定 | 站点/公众号 → 源候选池(D5) |
| **B3 定向核查搜索** | 事件粗筛通过后自动触发 | 事件单位名+关键要素 | `"单位名" + 事件词`、`site:官网域名`、公众号账号限定 | 补充来源(官方声明/公告)→ 挂接事件;可信度升级依据 |
| **B4 回访搜索** | T+30/90/180/365 任务到期 | 单位名/USCC + 回访目标 | `"单位名" × {处罚,判决,中标,招标,整改}` 在信用中国/裁判文书/政采网/搜索引擎 | 回填 L4/D2/D3 → 合并入原事件(禁新建) |
| **B5 监控名单搜索** | 按 watch 对象的频率(A/B/C) | watch_target(单位/产品/攻击组织/专题)+ 别名 | `对象名(含别名)× 事件词`,时间限定=上次水位线以来 | 命中 → 事件流水线;对象关联事件自动挂 related |
| **B6 同款预警搜索** | 事件含 affected_products 时触发 | 涉事产品名+版本 | `"产品名" × {漏洞,被利用,用户,案例,中标}`;政采网查该产品历史中标单位 | 潜客名单 → lead(same_product);产品自动加入 B5 监控 |
| **B7 突发热点搜索** | 热搜/热榜关键词命中安全词 | 热点词条 | 词条原文 + 事件词扩展,近 24h,全引擎 | 即时候选文档;触发相关源即时抓取一轮 |
| **B8 对标漏报搜索** | 月度基准对标发现漏报 | 漏报事件摘要 | 摘要关键要素重构查询,不限时间 | 补录事件;漏报归因(缺源/词盲区/误杀)→ 词表与源整改 |

**行为间的联动关系**(逻辑闭环):

```
B1/B7 发现事件 ──→ B3 定向核查(升可信度)──→ 发布
                                  │
        事件含涉事产品 ──→ B6 同款预警 ──→ 产品进 B5 监控
        事件金额未落地 ──→ B4 回访(按 T+N 再来)
        重点单位/组织  ──→ 人工/自动加入 B5 监控名单
B2 持续扩渠道;B8 月度兜底查漏 ──→ 反哺 B1 词表与源池
```

## 2. 十项通用能力(C1–C10)

| # | 能力 | 规格 |
|---|---|---|
| **C1 查询构造器** | 词组组合展开、`site:` 限定、平台时间参数映射、排除词注入、账号/案由等平台专有限定;查询串模板按适配器声明 |
| **C2 翻页与截断记录** | 每查询默认翻 3 页(可配),命中仍饱和时**记录截断事实**(禁止无声截断,漏报归因需要);翻页间随机延时 |
| **C3 跳转还原** | 搜索引擎跳转链(百度/搜狗中间链)必须还原为真实 URL 后再做归一化与发布方提取——否则源发现全记到搜索引擎域名上,整条 D1 通道报废 |
| **C4 水位线增量** | 每 (源,查询) 记录上次执行时间与已见 URL(`search_watermark`),下次只处理新增;时间限定参数取水位线以来,避免重复拉取与配额浪费 |
| **C5 结果双用途** | 每条结果固定两路:①URL→抓取流水线(事件);②发布方→源发现证据(渠道)。任何 B 类行为都不允许只取其一 |
| **C6 配额与频控** | 每源每日查询预算(keyword_matrix.query_budget);请求间随机延时+UA/代理池轮换;结果为空的组合自动降频 |
| **C7 反爬降级链** | 正常请求 → 换代理重试 ≤3 → 出验证码则该源冷却 30min → 连续失败转 manual_assist(生成查询链接清单待人工),**全程告警但不静默丢任务** |
| **C8 链接永久化** | 搜狗微信返回的是临时链,必须当刻访问并解析出 `mp.weixin.qq.com` 永久链接再入库存档;其他平台短链一律展开 |
| **C9 命中率统计与词表进化** | keyword_run 统计每查询组合的 结果数/新文档数/入库转化数;月度自动产出"低效词淘汰+建议新词(从入库事件标题高频词反推)"报告,人工确认后发新版词表 |
| **C10 可回放审计** | 每次搜索记录:查询串、平台、时间、结果清单快照——漏报争议时可回放"当时搜没搜到" |

## 3. 本次查验补上的逻辑缺口

| 缺口 | 问题 | 补法 |
|---|---|---|
| ① **监控名单机制缺失(最大缺口)** | 原设计搜索都是"泛关键词"或"事件触发",没有**持续盯防特定对象**的机制:重点行业头部企业、已涉事单位的后续、高危产品、活跃勒索组织——这些对象值得按名单主动搜,而不是等关键词撞上 | 新增 B5 + `watch_target` 表(kind: org/product/attacker_group/topic,含别名数组、频率、有效期);事件发布时自动建议入名单(涉事单位→B4 联动、涉事产品→B6 联动、攻击组织) |
| ② 跳转链未还原 | 搜索结果 URL 是引擎中间链,直接入库则去重失效、源发现记错主体 | C3 定为硬能力,抓取前强制还原 |
| ③ 无水位线 | 同一查询每天全量重拉,浪费配额且重复文档冲击流水线 | C4 + `search_watermark` 表 |
| ④ 微信临时链 | 搜狗微信链接数小时失效,事后无法访问原文 | C8:当刻解析永久链+当刻存档(与 7.3 呼应) |
| ⑤ 无声截断 | 翻页深度有限,命中饱和时漏掉的部分无人知晓,漏报归因无从做起 | C2:截断必须记录,月度对标时纳入归因 |
| ⑥ 境外源无英文词 | DataBreaches/Ransomware.live 等境外源用中文矩阵搜不到涉华内容 | keyword_matrix 增加 english_terms,仅作用于境外源 |

## 4. 数据落位

- `watch_target`(新表):监控对象与频率、别名、关联事件、有效期;
- `search_watermark`(新表):(源, 查询哈希) → 上次执行时间;
- `keyword_run` 增加 `watch_target_id`(B5 执行记录归因到对象);
- 配置:`keyword_matrix.yaml` 增加 `english_terms` 与 `max_pages_per_query`。
