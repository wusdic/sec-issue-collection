# 采集软件详细设计(v0.1)

> 上游依据:`docs/需求与解决方案.md`(模块 M1–M11)、`schema/event.schema.json`、`schema/dictionaries.yaml`。
> 本文档给出技术选型、服务划分、数据库设计、API 接口、流水线逻辑与部署方案,可直接开工。
> 配套文件:`design/schema.sql`(建库 DDL)、`config/seed_sources.yaml`(种子源)、`config/keyword_matrix.yaml`(关键词矩阵)。

---

## 1. 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 语言/框架 | Python 3.12 + FastAPI | 爬虫、LLM、数据处理生态最全;一期规模小,单语言栈降低维护成本 |
| 任务队列 | Celery + Redis | 抓取/存档/抽取/回访全部异步化;beat 做定时调度 |
| 浏览器渲染 | Playwright(Chromium) | 动态页渲染、整页/分段截图、MHTML 导出,一个引擎覆盖 7.3 全部存档层级 |
| 主库 | PostgreSQL 16 + pgvector | 事件 JSONB + 结构化查询列;pgvector 做事件语义去重召回 |
| 对象存储 | MinIO(S3 兼容) | 原文存档 WORM(对象锁)+ 双副本;后续可平移公有云 OSS |
| 全文检索 | PG `tsvector`(一期)→ Elasticsearch(量大后) | 一期日增几十篇,PG 足够,少一个组件 |
| LLM | OpenAI 兼容接口抽象层,可插拔(国内部署推荐 Qwen/DeepSeek 系;也可接 Claude API) | 粗筛与结构化抽取;接口抽象保证可换模型 |
| 前端 | Vue3 + Element Plus | 复核台/看板等管理端 |
| 部署 | Docker Compose(一期单机) | 方案 14 节结论:重点在流程不在算力 |

## 2. 服务划分与进程拓扑

```
api          FastAPI:全部 REST 接口 + 前端静态资源
worker-crawl Celery 队列 crawl:页面型/查询型适配器执行、M11 存档
worker-nlp   Celery 队列 nlp:粗筛、抽取、simhash、embedding
worker-ops   Celery 队列 ops:回访任务、存档校验、报表生成、KPI 计算
beat         Celery beat:调度(源抓取频率、T+N 回访、月度校验/对标)
postgres / redis / minio
```

一个代码仓库、按 Celery 队列拆 worker,不做微服务。

## 3. 数据库设计

完整 DDL 见 `design/schema.sql`。表清单与关键设计:

| 表 | 模块 | 说明 |
|---|---|---|
| `source` | M1/M10 | 源注册表;`kind`=page/query,`lifecycle`=candidate/trial/active/degraded/retired;贡献度计数列 |
| `source_metric_daily` | M1 | 每源每日:抓取数/新文档数/失败数,驱动 A/B/C 频率建议与退役建议 |
| `crawl_run` | M2 | 每次抓取执行记录(源、起止、状态、错误) |
| `raw_document` | M2/M3 | 原始文档:URL(原始+归一化+最终)、标题、发布方、发布时间、正文文本、simhash、粗筛结果与得分、快照 ID |
| `doc_cluster` | M2 | 同稿簇:首发文档、成员数、最早发布时间(热度信号) |
| `archive_manifest` | M11 | 存档:层级 L-A/B/C/D、图片/附件/截图计数、manifest sha256、存储路径、校验记录 |
| `event` | M4/M5 | 事件主表:`payload` JSONB(通过 event.schema.json 校验)+ 高频查询列(时间/行业/地区/严重度/攻击类型数组等,入库触发器从 payload 同步)+ `embedding` vector |
| `event_source` | M5 | 事件↔文档/来源关联:credibility、supports_fields |
| `event_change_log` | M9 | 字段级变更审计 |
| `review_task` | M5 | 复核流转:extracted→first_review→(金额双签)second_review→published/rejected |
| `followup_task` | M6 | 回访:类型 T30/T90/T180/T365/manual、到期日、检索建议、结果 |
| `keyword_set` / `keyword_run` | M2/M10 | 关键词矩阵版本与每日查询执行记录 |
| `benchmark_batch` / `benchmark_item` | M10 | 月度基准清单与逐条命中/漏报归因 |
| `lead` | M8 | 线索:事件、目标单位、评分、窗口阶段、建议产品、下发/跟进状态(CRM 回流) |
| `dictionary_release` | M9 | 词表版本(dictionaries.yaml 整包入库,发布制) |
| `app_user` / `audit_log` | M9 | 账号角色(录入/复核/分析师/只读)与操作审计 |

关键约束(在 DB 层强制,呼应方案硬规则):

- `event` 发布触发器:`confirmed` 金额通道非空 ⇒ 必须存在 credibility ∈ {S1,S2} 的关联来源,否则拒绝发布(方案 11.4 口径回归的库级兜底);
- `archive_manifest.status='L-D'` 的文档不可被引用为已发布事件的唯一来源;
- `raw_document.url_normalized` 唯一索引(10.1 URL 去重);
- 存档桶开启对象锁(WORM),DB 内只存路径与哈希。

## 4. 核心流水线逻辑

### 4.1 抓取与存档(M2+M11,queue: crawl)

```
beat 按 source.tier 触发 → crawl_run 开始
页面型: 列表页解析 → 新 URL(归一化查重) → Playwright 渲染
查询型: keyword_set 展开为查询串(带时间限定) → 搜索结果解析 → 同上
渲染成功后同一浏览器会话内完成 M11 存档:
  ① 保存 DOM HTML + 抓取页内 <img>/附件链接,带 Referer/UA 下载至本地 → L-A
  ② 失败项 → page.pdf() + CDP 导出 MHTML → L-B
  ③ 仍失败 → 整页截图;页高 > 3×视口 → 按视口高 90% 步进分段截图 part_NNN.png + 全页 full.png → L-C
  ④ 全失败 → L-D + 人工补档队列
写 manifest.json(逐文件 sha256)→ MinIO(WORM) → archive_manifest
正文提取(trafilatura,失败退 readability)→ raw_document → 投递 nlp 队列
```

### 4.2 粗筛与同稿聚类(M3/10.2,queue: nlp)

```
simhash(64bit) 与近 30 天文档比对,海明距离 ≤3 → 并入 doc_cluster(仅首发走后续流程)
LLM 粗筛:判定 {是否国内安全事件候选, 置信度, 理由}
  ≥0.7 → 进抽取;0.4–0.7 → 人工快筛队列;<0.4 → screened_out(留 90 天)
```

### 4.3 结构化抽取(M3)

- 单次 LLM 调用,输出严格对齐 `event.schema.json`(JSON mode + 本地 jsonschema 校验,失败带错误重试 ≤2 次);
- prompt 注入当前词表版本 + **金额三态判别规则**:
  - 命中"要求/索赔/勒索/拟处罚/预计/或将/传" → 只允许写 `claimed`;
  - 命中"判决/裁定/处罚决定书/公告披露/年报确认" → 写入 `confirmed` **候选**,复核台强制人工确认;
  - 抽取器**永远无权**直接定稿 confirmed 通道;
- 每个抽取值附 `source_span`(原文片段定位),复核台点字段高亮原文依据。

### 4.4 事件去重与合并(M4/10.3)

```
候选事件 → 单位归一(名称标准化;工商联查得 USCC 则回填)
指纹查询: (org_key, attack_type∩, occurred±14d) 精确碰撞
未命中 → embedding(事件摘要) pgvector 余弦 top5 > 0.88 → 疑似合并,转人工
合并: 保留全部来源,字段按可信度择优,写 event_change_log
```

### 4.5 复核与发布(M5)

状态机:`extracted → first_review → [含金额确认通道时 second_review] → published | rejected`。
复核台强规则(前后端双重校验):原文含"要求/索赔"字样而金额被填入 confirmed → 阻断保存;
confirmed 无 S1/S2 来源 → 阻断发布;B7/B9 未填 → 阻断发布(可填"未披露/不明")。

### 4.6 回访(M6,queue: ops)

发布时按方案第 8 节条件生成 T+30/90/180/365 followup_task;
到期任务附"一键检索包":单位名 × {处罚,判决,中标,招标} 在裁判文书网/信用中国/政府采购网的查询链接与查询型适配器结果;
回填走同一抽取+复核流程,合并入原事件(禁止新建)。

### 4.7 线索与报表(M8,queue: ops)

发布/更新事件 → 规则引擎(手段×入口×根因×后果 YAML 规则)算 D5 → 生成/更新 lead
(评分公式与窗口阶段见方案 12.3;窗口阶段由 beat 每日重算,过期降级);
月报/对标/KPI 为 ops 队列定时任务,产物落 MinIO 供下载。

## 5. REST API 设计(前缀 /api/v1)

| 资源 | 端点 | 说明 |
|---|---|---|
| 源 | `GET/POST /sources`,`PATCH /sources/{id}`,`POST /sources/{id}/trial`、`/promote`、`/retire`,`GET /sources/{id}/metrics` | M1 生命周期与指标 |
| 文档 | `GET /documents?status=`,`GET /documents/{id}`,`POST /documents/{id}/rescreen` | 原始文档与人工快筛 |
| 存档 | `GET /archives/{snapshot_id}`(manifest),`GET /archives/{snapshot_id}/view`(离线原文查看器),`POST /archives/{snapshot_id}/reverify` | M11 |
| 事件 | `GET /events?industry=&province=&severity=&attack_type=&from=&to=`,`GET/POST /events/{id}`,`POST /events/{id}/merge`,`GET /events/{id}/changelog` | 事件 CRUD 走 schema 校验 |
| 复核 | `GET /review/queue?stage=`,`POST /review/{task_id}/submit|approve|reject` | 状态机流转 |
| 回访 | `GET /followups?due=`,`POST /followups/{id}/complete`,`GET /followups/{id}/search-pack` | 一键检索包 |
| 关键词 | `GET/POST /keyword-sets`,`POST /keyword-sets/{id}/publish` | 版本化发布 |
| 对标 | `POST /benchmarks`(上传基准清单),`GET /benchmarks/{id}/report` | 漏报归因 |
| 线索 | `GET /leads?stage=&min_score=`,`POST /leads/{id}/dispatch`,`POST /leads/{id}/feedback` | CRM 下发与回流 |
| 报表 | `GET /reports/heatmap|loss|regulatory|controls|products|competition`,`GET /reports/monthly/{yyyymm}` | 12.1 看板数据 |
| 词表 | `GET /dictionaries/current`,`POST /dictionaries/releases` | 发布制 |
| 系统 | `POST /auth/login`,`GET /kpi/dashboard`,`GET /audit-logs` | M9 |

鉴权:JWT + 角色(admin/analyst/reviewer/editor/readonly);全部写操作入 `audit_log`。

## 6. 适配器框架与种子源

适配器基类 `BaseAdapter`:`discover() -> [url]`(列表页/搜索结果解析)与 `fetch(url) -> RenderedPage`(渲染+存档钩子);
具体适配器只写解析逻辑,渲染/存档/去重由框架统一处理。
种子源 40 个见 `config/seed_sources.yaml`(含 URL、适配器类型、频率层级、可信度、反爬备注)。
需要账号/风控强的源(裁判文书网、部分搜索引擎)标注 `manual_assist: true`——一期允许"系统给出查询链接+人工粘贴结果"的半自动模式,不硬啃反爬(合规且省工期)。

## 7. 部署与配置

- `docker-compose.yml`:api / worker-crawl / worker-nlp / worker-ops / beat / postgres / redis / minio / frontend;
- 关键环境变量:`LLM_BASE_URL/LLM_API_KEY/LLM_MODEL`(OpenAI 兼容)、`EMBEDDING_MODEL`、`ARCHIVE_BUCKET`(开对象锁)、`CRAWL_PROXY_POOL`;
- 备份:PG 每日全量 + WAL;MinIO 双副本 + 月度抽检(ops 任务)。
