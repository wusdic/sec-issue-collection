# 通用信息搜索框架(v0.1)

> 把安全事件库的搜索逻辑**泛化为领域无关的信息搜集框架**:
> 任何"信息需求"都实例化为一份配置(需求画像),由同一套引擎执行
> **持续搜索 → 结构化 → 持续更新 → 覆盖度量** 的闭环。
> 安全事件库是本框架的第一个实例;政策动态库、招标全量集(方案 12.4)是第二、三个实例。
> 配套模板:`config/need_profile.template.yaml`。

---

## 1. 核心抽象:一切从"信息需求"出发

**信息需求(Information Need)= 你想持续知道的一类事。** 每个需求由六要素定义:

| 要素 | 回答的问题 | 安全事件库中的对应 |
|---|---|---|
| ① 目标记录 Schema | 我要的信息长什么结构?哪些字段支撑决策? | event.schema.json |
| ② 领域词表 | 用什么词能搜到它?怎么分类统计? | dictionaries.yaml + keyword_matrix |
| ③ 来源策略 | 谁会发布它?各来源可信度如何分级? | seed_sources + S1–S4 |
| ④ 更新策略 | 记录会怎么演化?哪些字段滞后出现?多久回访? | 生命周期 + T+N 回访 |
| ⑤ 质量规则 | 什么算"确认"?什么只能算"声称"?入库门槛是什么? | 金额三态 + 最少字段集 |
| ⑥ 输出物 | 谁用?什么形式?多久一次? | 线索单/月报/看板 |

**框架的不变量**(所有需求实例共享,不可裁剪):
字段级来源绑定、原文完整存档(降级链)、三层去重、来源可信度分级、
断言三态(声称/第三方估算/权威确认——金额三态的泛化)、记录生命周期与变更日志、覆盖度量。

## 2. 通用流水线(与需求无关的执行骨架)

```
┌──────────────────────── 需求画像(need profile)驱动 ────────────────────────┐
│                                                                            │
│ 源发现 ⇄ 源池 → 采集+存档 → 去重(URL/文档/记录) → 粗筛 → 结构化抽取        │
│                                                        ↓                   │
│ 覆盖对标 ←── 应用输出 ←── 生命周期更新 ←── 复核发布 ←── 实体归一/合并      │
│     └──────────── 漏报归因反哺词表与源池(进化回路) ────────────┘           │
└────────────────────────────────────────────────────────────────────────────┘
```

每个环节读需求画像中的对应配置执行;新增一个信息需求 = 写一份画像,不改引擎。

## 3. 八类搜索行为的泛化(G1–G8)

安全事件库的 B1–B8(见《搜索行为逻辑与能力规范》)泛化为任何信息需求通用的八类:

| # | 泛化行为 | 通用逻辑 | 安全库实例 | 其他需求示例(招标情报) |
|---|---|---|---|---|
| **G1 发现搜索** | 领域词表 × 时间限定,周期性搜"新出现的目标记录" | B1 事件发现 | "安全服务 中标公告 近24h" |
| **G2 渠道搜索** | 搜"谁在持续发布这类信息",产出新源候选 | B2 找源 | "招标信息 聚合 网站" |
| **G3 核实搜索** | 对新记录定向搜权威来源,提升断言等级 | B3 定向核查 | 中标价与官方公告核对 |
| **G4 追踪搜索** | 对已知记录按滞后字段的节奏定期回访补全 | B4 回访(罚款/判决) | 中标→合同签订→验收公告 |
| **G5 名单监控** | 对高价值对象(实体/产品/主体)按名单持续盯防 | B5 watch_target | 重点客户的所有新招标 |
| **G6 关联扩展** | 从记录属性出发横向搜同类/相关记录 | B6 同款预警 | 同一采购人历史项目 |
| **G7 突发触发** | 外部热点信号命中领域词,即时加密度搜索 | B7 热搜触发 | 政策发布→相关招标潮 |
| **G8 兜底对标** | 用独立基准清单反查库内召回率,漏报归因 | B8 月度对标 | 与中标数据商月度比对 |

**行为联动是框架内置的**(不随需求变化):G1/G7 产新记录 → G3 核实 → G4 按滞后字段排期追踪;
记录属性触发 G6 扩展、对象沉淀进 G5 名单;G2 扩渠道、G8 兜底,共同反哺 G1 的词表与源池。

## 4. 通用能力层(C1–C10,直接复用)

《搜索行为逻辑与能力规范》第 2 节的十项能力本身就是领域无关的,全部平移:
查询构造器 / 翻页与截断记录 / 跳转还原 / 水位线增量 / 结果双用途(记录+渠道)/
配额频控 / 反爬降级链 / 链接永久化 / 命中率统计与词表进化 / 可回放审计。

唯一随需求变化的是**查询构造的词表来源**和**结果判定的粗筛标准**——均由需求画像提供。

## 5. 泛化的质量与更新模型

- **断言三态(泛化自金额三态)**:任何关键断言分为 `声称(claimed)/第三方估算(estimated)/权威确认(confirmed)`,
  确认通道必须挂该需求定义的权威来源等级(安全库=S1/S2;招标库=官方采购网原公告);
- **记录生命周期(泛化自事件生命周期)**:`draft → published → monitoring → closed`,
  需求画像声明哪些字段是"滞后字段"及其典型滞后周期,引擎据此自动生成 G4 追踪任务;
- **实体归一是跨需求共享的**:单位(统一社会信用代码)、产品、人物等实体库全局唯一,
  多个需求实例的记录挂接同一实体——这是跨库交叉分析的基础
  (例:事件库"某医院被勒索" × 招标库"该医院 3 个月后采购灾备" 自动关联);
- **覆盖度量是强制项**:每个需求实例必须声明至少一个独立基准(官方汇总/商业数据/人工抽样),
  没有基准的需求无法度量"全不全",不允许上线。

## 6. 多需求并行的架构含义

- **源池共享**:一个源可服务多个需求(政府采购网同时供事件回访、招标全量、企业画像);
  源注册表加 `serves_needs[]`,抓取一次、多需求分发,不重复抓取;
- **存档共享**:原文存档以文档为单位全局唯一,多需求引用同一快照;
- **词表与画像隔离**:每个需求的 schema/词表/质量规则独立版本化,互不干扰;
- **复核队列按需求分流**:复核人按需求域配置权限与专业分工;
- **调度按需求加权**:抓取与查询配额在需求间按优先级分配,新需求上线不挤爆存量需求。

对应实现:`need_profile` 表(需求画像注册)+ 现有各表增加 `need_id` 维度
(raw_document/记录表/keyword_run/watch_target/benchmark 等);一期安全事件库可先以
单需求运行,表结构预留 need_id 即可,不增加当期工作量。

## 7. 新需求实例化清单(上线一个新信息需求的标准动作)

1. 填写需求画像(`config/need_profile.template.yaml`):六要素齐备;
2. 定义目标记录 JSON Schema + 领域词表 v1(枚举可统计,留"未披露"值);
3. 配置种子源与可信度分级标准、G1 发现词表、G2 找源词表;
4. 声明滞后字段与追踪节奏(G4)、名单对象类型(G5)、扩展规则(G6)、突发信号(G7);
5. 声明覆盖基准(G8)与月度对标方式;
6. 定义输出物与消费方(报表/线索/API 订阅);
7. 试运行 2 周 → 对首批记录做人工全检 → 校准词表与粗筛阈值 → 正式上线。
